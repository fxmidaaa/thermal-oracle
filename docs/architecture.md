# ThermalOracle — System Design (v1)

Дата: 2026-06-11. Статус: **принят без правок** (ревью владельца продукта
2026-06-11). Решения по открытым вопросам зафиксированы в [adr/](adr/):
Windows-first (ADR-0001), управляемая калибровка в v2 (ADR-0002).

ThermalOracle — micro-SaaS предиктивного мониторинга деградации термоинтерфейсов
(термопаста, жидкий металл, phase-change материалы типа Honeywell PTM7950) для
высокопроизводительных ноутбуков и ПК. Продукт отвечает на вопрос: «насколько
выросло тепловое сопротивление моей системы охлаждения со временем и когда пора
менять термоинтерфейс».

## 1. Обзор системы

```
 [Агент Windows]──┐
  LibreHardwareMonitor│  HTTPS, gzip-батчи раз в 30с,
  localhost:8085      │  device token, идемпотентные batch_id
 [Агент macOS]────┤
  sudo powermetrics   ▼
            ┌────────────────────────────┐
            │ FastAPI  POST /v1/telemetry│── Pydantic → дедуп → bulk insert
            └────────────┬───────────────┘
                         ▼
            ┌────────────────────────────┐
            │ PostgreSQL + TimescaleDB   │  raw(1с) → CAgg 1m → CAgg 1h
            │ compression / retention    │  + производные таблицы аналитики
            └────────────┬───────────────┘
                         ▲▼
            ┌────────────────────────────┐
            │ Analytics Worker (отд.процесс)│ idle→T_ambient; окна нагрузки→Rth;
            │ APScheduler, инкрементальные │ тренды→health score→прогноз→алерты
            │ и дневные джобы              │
            └────────────────────────────┘
                         ▲
 [Дашборд / CLI юзера]───┘  FastAPI GET /v1/devices|health|rth-trend (user JWT)
```

### 1.1 Ключевые архитектурные принципы

1. **Raw неизменяем, аналитика воспроизводима.** Сырая телеметрия пишется как
   есть (включая подозрительные значения — они лишь флагуются). Вся фильтрация
   шума происходит на этапе анализа. Каждая производная таблица — детерминированная
   функция от raw + `model_version`, поэтому любой отрезок можно пересчитать при
   улучшении алгоритмов или при поступлении опоздавших данных.
2. **Идемпотентный ingest, семантика at-least-once.** Агент может слать один и
   тот же батч сколько угодно раз (ретраи, реплей оффлайн-буфера) — на сервере
   ровно один результат. Дедупликация — трёхслойная (раздел 4.3).
3. **Тонкий агент, умный сервер.** Агент только собирает и буферизует; детекция
   окон, фильтрация и математика — на сервере. Иначе каждое улучшение алгоритма
   требовало бы обновления агентов у всех пользователей и порождало version skew.
4. **Без лишней инфраструктуры.** На целевом масштабе (тысячи устройств) очередь
   сообщений не нужна: 1000 устройств × 1 батч/30с ≈ 33 запроса/с ≈ 1000 строк/с —
   это копейки для прямой вставки в Timescale. HTTP-контракт позволяет вставить
   Redis Streams/NATS позже без изменения агента.
5. **Долгоживущий актив — не сырьё, а выводы.** Raw хранится 30 дней, а компактные
   `rth_windows`, `ambient_estimates`, `device_health` — бессрочно. Именно они
   и есть продукт.

## 2. Локальный агент (Python CLI)

### 2.1 Сбор данных

Частота сэмплирования — 1 Гц, по монотонным часам (`time.monotonic` для
интервалов, `time.time_ns` UTC для меток). Каждый сэмпл:

| Поле | Тип | Источник Win (LHM) | Источник macOS (powermetrics) |
|---|---|---|---|
| `ts` | UTC, мс | системные часы | системные часы |
| `cpu_temp` | °C | Sensor «CPU Package» / «Core (Tctl/Tdie)» | smc / thermal sampler |
| `gpu_temp` | °C | узел GPU → Temperature | для iGPU = die CPU |
| `cpu_power` | Вт | Sensor «CPU Package» (Power) | cpu_power sampler (мВт→Вт) |
| `gpu_power` | Вт | узел GPU → Power | gpu_power sampler |
| `fan_rpm` | RPM | max по всем Fan-сенсорам | smc sampler |
| `process` | str | foreground-процесс (Win32 API) | NSWorkspace frontmost |

Если сенсор не ответил — поле = `null`. **Запрещено повторять последнее
значение**: «замороженный» сенсор неотличим от реального плато и портит анализ.

**Windows.** Ring-0 доступ без админ-прав закрыт, поэтому агент парсит JSON
веб-сервера LibreHardwareMonitor (`http://localhost:8085/data.json`). Грабли,
которые закладываем в дизайн коллектора:
- значения приходят строками с локалью («45,3 °C» в ru-RU) — парсер обязан
  обрабатывать запятую как десятичный разделитель;
- дерево сенсоров и их имена зависят от вендора и версии LHM. Поэтому при
  регистрации выполняется `thermal-agent detect-sensors`: агент показывает
  найденные кандидаты, выбранные пути сохраняются в локальный конфиг **и**
  отправляются на сервер в `devices.sensor_map` — бэкенд всегда знает, из какого
  сенсора пришла каждая серия;
- глюки чтения (нули, 255, залипшие значения) — агент ставит `null`/флаг,
  диапазонная валидация дублируется на сервере.

**macOS.** Агент спавнит `sudo powermetrics -i 1000 -f plist --samplers
cpu_power,gpu_power,smc,thermal` и стримово парсит plist-блоки. `sudo` — только
на сам powermetrics через документированное NOPASSWD-правило в sudoers; агент
не работает под root. На Apple Silicon температура кристалла доступна
ограниченно (часто только thermal pressure) — поэтому вводим **capabilities**:
при регистрации устройство декларирует `{has_cpu_temp, has_gpu_power,
has_fan_rpm, ...}`, и аналитика включает только те модули, для которых хватает
данных (минимум для Rth: температура + мощность).

### 2.2 Буферизация и отправка

- Сэмплы складываются в локальный SQLite-спул (ring buffer, по умолчанию 24 ч —
  конфигурируемо; решение Шага 3 /
  ~50 МБ) — переживает оффлайн, сон ноутбука и краш агента.
- Каждые 30 с шиппер собирает батч (≈30 сэмплов; после оффлайна — до 120),
  присваивает `batch_id = UUIDv7`, жмёт gzip и шлёт `POST /v1/telemetry`.
- Ретраи с экспоненциальным backoff + jitter (1с → 2 → 4 → … → cap 5 мин).
  Сэмплы удаляются из спула **только после 2xx** (включая ответ «duplicate»).
  Ответ 503/429 с `Retry-After` уважается — backpressure от сервера.
- В батче передаётся `sent_at` — сервер фиксирует clock skew устройства.

### 2.3 Приватность

Имя активного процесса — чувствительные данные (раскрывают активность
пользователя). Режим в конфиге агента: `full` (как есть) | `hashed`
(SHA-256, сервер сопоставляет только с известным словарём игр/бенчмарков) |
`off`. По умолчанию `full`, явно проговаривается при регистрации.

### 2.4 CLI

`thermal-agent register --pairing-code XXX` · `detect-sensors` · `run`
(foreground; для автозапуска — Task Scheduler / launchd, инструкции в доке) ·
`status` (последняя отправка, размер спула, здоровье сенсоров).

## 3. Wire-контракт ingest

```jsonc
POST /v1/telemetry   (Authorization: Bearer <device_token>, Content-Encoding: gzip)
{
  "schema_version": 1,
  "batch_id": "0190b5c4-…",          // UUIDv7 — ключ идемпотентности
  "sent_at": "2026-06-11T10:00:30.120Z",
  "agent_version": "0.3.1",
  "samples": [                        // 1..120 элементов
    {"ts": "...", "cpu_temp": 47.5, "gpu_temp": 41.0,
     "cpu_power": 8.2, "gpu_power": 1.1, "fan_rpm": 2300, "process": "chrome.exe"},
    ...
  ]
}
→ 200 {"accepted": 28, "duplicates": 2, "rejected": 0}
```

Pydantic-валидация: строгие типы, длина массива, ISO-метки. Диапазонные
проверки (**не отклоняют батч**, а обнуляют поле и ставят бит во `flags`):
температуры −20…120 °C, мощности 0…400 Вт, RPM 0…10 000. Сэмплы с
`ts > now+120с` отбрасываются (сломанные часы), с `ts < now−7д` — дропаются с
метрикой (за горизонтом реанализа, см. 4.4). Бэкенд поддерживает
`schema_version` N и N−1.

## 4. База данных (PostgreSQL + TimescaleDB)

### 4.1 Реляционное ядро (обычные таблицы)

```sql
users (
  id uuid PK, email citext UNIQUE, password_hash text,
  plan text DEFAULT 'free', created_at timestamptz
)

devices (
  id uuid PK, user_id uuid FK→users,
  name text, platform text CHECK (platform IN ('windows','macos')),
  device_class text CHECK (device_class IN ('laptop','desktop')),  -- пресеты порогов
  cpu_model text, gpu_model text,       -- для T_throttle и когортных сравнений (v2)
  capabilities jsonb,                   -- {"has_fan_rpm": true, ...}
  sensor_map jsonb,                     -- выбранные пути сенсоров LHM/SMC
  analysis_overrides jsonb,             -- точечные переопределения порогов
  agent_version text, timezone text,
  created_at, last_seen_at timestamptz
)

device_tokens (
  id uuid PK, device_id FK→devices,
  token_prefix text,                    -- "to_ab12" для быстрого lookup
  token_hash bytea,                     -- SHA-256; секрет показывается один раз
  created_at, last_used_at, revoked_at timestamptz
)

-- Журнал обслуживания: режет историю устройства на «эпохи».
-- Замена пасты обнуляет деградацию — тренд считается только внутри эпохи.
maintenance_events (
  id bigserial PK, device_id FK, ts timestamptz,
  kind text CHECK (kind IN ('repaste','cleaning','repad','fan_curve_change',
                            'undervolt_change','hw_change','regime_change')),
  tim_type text CHECK (tim_type IN ('paste','liquid_metal','ptm7950',
                                    'stock','unknown')),
  source text DEFAULT 'user',           -- 'user' | 'changepoint_suggested'
  note text
)
-- 'regime_change' создаёт только update_trends (source='changepoint_suggested',
-- дедуп ±3 дня): CUSUM увидел ступеньку — «подтвердите, что это было».
-- Эпоху НЕ режет, пока пользователь не оформит настоящее событие через API.

processes (id serial PK, name text UNIQUE)   -- словарь имён процессов

alerts (
  id bigserial PK, device_id FK, kind text, severity text,
  created_at timestamptz, payload jsonb, acked_at timestamptz
)
```

Multi-tenancy: вся телеметрия ключуется `device_id`; принадлежность
device→user проверяется на API-слое (devices — маленькая, кэшируется in-process
с TTL 60с). `user_id` в гипертаблицы **не** денормализуем — ни один запрос не
читает телеметрию мимо конкретного устройства. Row-Level Security осознанно
отложена: при единственном сервисе-потребителе app-level проверки достаточно;
RLS добавим, когда к базе подключатся внешние читатели (BI, интеграции).

### 4.2 Гипертаблицы и rollup

```sql
telemetry_raw (
  device_id uuid NOT NULL,
  ts timestamptz NOT NULL,
  cpu_temp real, gpu_temp real,         -- NULL = сенсор не ответил
  cpu_power real, gpu_power real,
  fan_rpm int,
  process_id int REFERENCES processes(id),
  flags smallint NOT NULL DEFAULT 0,    -- бит 0: out-of-range, бит 1: clock-suspect…
  UNIQUE (device_id, ts)                -- включает партиционирующую колонку — ок
);
SELECT create_hypertable('telemetry_raw', 'ts',
                         chunk_time_interval => interval '1 day');
-- Компрессия: segmentby = device_id, orderby = ts DESC.
-- compress_after = '7 days'  ← РОВНО горизонт приёма опоздавших данных:
--   вставка в сжатые чанки дорогая, поэтому всё «опоздавшее» должно успеть
--   приехать до компрессии. Связка этих двух констант — инвариант системы.
-- Retention: drop_chunks через 30 дней.

-- Журнал батчей: идемпотентность + аудит + диагностика clock skew.
ingest_batches (
  device_id uuid, batch_id uuid,
  received_at timestamptz, sent_at timestamptz,
  sample_count int, dup_count int, rejected_count int,
  agent_version text,
  PRIMARY KEY (device_id, batch_id)
);
-- ОБЫЧНАЯ таблица, не гипертаблица (уточнение при имплементации, миграция
-- 0002): уникальный индекс гипертаблицы обязан включать партиционирующую
-- колонку времени, что сломало бы семантику PK (device_id, batch_id) — основу
-- идемпотентности слоя 1. Чистка — ночной DELETE по received_at (индекс есть)
-- в worker'е; при >500 активных устройств объём DELETE пересмотреть.
-- Корректность от чистки не зависит: слой 2 (UNIQUE device_id, ts) —
-- самодостаточный backstop.
```

Continuous aggregates (каскадом, hierarchical CAggs):

```sql
telemetry_1m:  time_bucket('1 minute'), device_id,
               avg/max(cpu_temp), avg/max(gpu_temp),
               avg/max(cpu_power), avg/max(gpu_power),
               avg/max(fan_rpm), count(*) AS n
               -- refresh-политика: каждую минуту, lag 2 мин; retention 180 дней
telemetry_1h:  CAgg поверх telemetry_1m; retention 2 года; сжатие
```

Сводная политика хранения:

| Уровень | Гранулярность | Хранение | Назначение |
|---|---|---|---|
| `telemetry_raw` | 1 с | 30 дн (сжатие после 7) | пересчёт аналитики, зум в дашборде, отладка |
| `telemetry_1m` | 1 мин | 180 дн | графики дашборда |
| `telemetry_1h` | 1 ч | 2 года | долгие тренды температур |
| `rth_windows`, `ambient_estimates`, `device_health`, `alerts` | событийная | бессрочно | основной актив продукта |

**Оценка объёма.** Строка raw ≈ 60 Б полезных + ~35 Б оверхеда ≈ 95 Б. Ноутбук
работает ~8 ч/день → 28.8 тыс. строк ≈ 2.7 МБ/день/устройство до сжатия;
колоночная компрессия Timescale на такой телеметрии даёт 10–20× → **0.15–0.3
МБ/день**. 30 дней × 1000 устройств ≈ 5–9 ГБ сырого слоя. Вывод: один скромный
инстанс Postgres тянет тысячи устройств; узкое место не storage и не ingest
(1000 строк/с), а аналитические джобы — они параллелятся шардированием
`device_id % N_workers`.

**Retention и тарифы.** `drop_chunks` глобален для гипертаблицы — пер-тенантный
retention означал бы дорогие DELETE-джобы. Решение: физическое хранение у всех
одинаковое (щедрое), тарифы различаются числом устройств, глубиной выдачи API
и фичами, а не storage. Это осознанный трейд-офф, его не «чиним».

### 4.3 Дедупликация (три слоя)

1. **Батч-уровень (HTTP-ретраи).** `INSERT INTO ingest_batches … ON CONFLICT DO
   NOTHING`; если конфликт — батч уже обработан, отвечаем `200 {"status":
   "duplicate"}` не трогая данные. Агент трактует как успех и чистит спул.
2. **Строка-уровень (пересечение батчей).** После краша агент может переупаковать
   спул в батчи с новыми `batch_id`, но теми же сэмплами. `UNIQUE (device_id,
   ts)` + `INSERT … ON CONFLICT DO NOTHING` гарантирует ровно одну строку на
   момент времени. Вставка батча и запись в `ingest_batches` — одна транзакция.
3. **Временные ворота.** Будущее (>+120с) — reject; старше 7 дней — drop с
   инкрементом метрики `dropped_too_late`. Метки времени — только UTC; skew
   фиксируется как `received_at − sent_at`.

Дубликаты не считаются ошибкой ни на одном слое — это штатный режим
at-least-once доставки.

### 4.4 Опоздавшие данные и пересчёт

Аналитика работает с watermark: инкрементальные джобы обрабатывают интервал
`[last_processed − 60с overlap, now − 2 мин]`. Если ingest принял сэмплы старше
watermark (дренаж оффлайн-спула), он пишет диапазон в `reprocess_queue
(device_id, range_start, range_end)` — ночной джоб пересчитывает окна и ambient
затронутых суток. Все производные строки апсертятся по натуральным ключам
(`device_id + window_start + domain` и т.п.), поэтому пересчёт идемпотентен.

## 5. Аналитический модуль

### 5.1 Физическая модель и почему фильтрация именно такая

Тепловое сопротивление junction→ambient:

```
Rth = (T_die − T_ambient) / P        [K/W]
```

Формула валидна **только в квазистационарном режиме**. При скачке мощности
температура релаксирует как `T(t) = T_∞ − (T_∞ − T₀)·e^(−t/τ)`, где τ связки
«кристалл + теплотрубки + радиатор» ≈ 5–20 с у ноутбука (у десктопных
башен — заметно больше). Отсюда требования ТЗ получают физический смысл:

- **всплеск < 3 с** — температура не успевает отреагировать, точка Rth была бы
  мусором → игнорируем;
- **стабильная нагрузка ≥ 15 с** — система успевает приблизиться к
  установившемуся состоянию (1–3 τ) → точка валидна;
- дополнительный гейт сверх ТЗ: **|dT/dt| в хвосте окна < 0.15 °C/с** —
  подтверждение фактического выхода на плато, а не только формальной
  длительности. Для `device_class='desktop'` пресет минимальной длительности
  поднимается до 30–45 с (большая тепловая масса).

Полное Rth раскладывается как `Rth_jc + Rth_TIM + Rth_heatsink→air(RPM)`.
Деградация термоинтерфейса растит слагаемое `Rth_TIM`, но измеряем мы только
сумму — поэтому конфаундеры (обороты вентилятора, уровень мощности, тип
нагрузки) надо контролировать стратификацией (5.5).

### 5.2 Оценка T_ambient

Прямого датчика комнатной температуры нет — используем температуру кристалла в
глубоком простое как прокси (спека: Power < 5 Вт).

Алгоритм (дневной джоб, по локальному дню устройства):

1. **Idle-эпизод** — непрерывный интервал, где скользящее среднее 30 с
   `P_cpu < 5 Вт` (мгновенные значения < 8 Вт), длительностью ≥ 15 мин.
2. **Отбрасываем первые 10 мин** эпизода: после нагрузки радиатор ещё тёплый
   (thermal soak-back), ранние минуты завышают оценку.
3. Оценка эпизода = **5-й перцентиль** сглаженной (медианный фильтр 5 с)
   `cpu_temp` по остатку эпизода. Низкий перцентиль, потому что любая фоновая
   активность смещает только вверх.
4. `T_amb(day)` = медиана оценок эпизодов, взвешенная длительностью.
   `confidence` растёт с суммой idle-минут и падает с разбросом оценок.
5. Санити: кламп 5…45 °C; выброс против медианы за 7 дней > ±8 °C — флагуется.
6. **Fallback:** если за день нет валидных эпизодов — carry-forward последней
   оценки с экспоненциальным затуханием confidence (×0.8/день). После 7 дней
   без калибровки все Rth-точки помечаются `low_confidence`, в UI — подсказка
   «оставьте машину в простое на 20 минут».

Документируемая систематика: `T_idle_die = T_room + P_idle·Rth ≈ T_room + 2–4 °C`.
Для **тренда** деградации постоянное смещение безвредно — нас интересует
изменение во времени, а не абсолютная точность. Зато сезонный ход комнатной
температуры (зима 20 °C / лето 28 °C) вычитается именно этим механизмом — без
него потепление в комнате маскировалось бы под деградацию пасты.

### 5.3 Детекция стабильных окон нагрузки (фильтрация шума)

Инкрементальный джоб (раз в 5 мин) гонит state machine по 1 Гц-серии мощности
каждого активного устройства, отдельно для доменов cpu/gpu:

```
                 P > 35 Вт                t_load ≥ 15 с
   ┌────────┐ ───────────────► ┌───────────┐ ──────────► ┌──────────┐
   │ BELOW  │                  │ CANDIDATE │             │  STABLE  │
   └────────┘ ◄─────────────── └───────────┘             └────┬─────┘
       ▲        P < 30 Вт дольше 3 с (грейс)                  │ закрытие окна
       └──────────────────────────────────────────────────────┘
Закрытие STABLE → эмит rth_window. Причины закрытия:
  • P < 30 Вт дольше 3 с (гистерезис: вход 35 Вт, выход 30 Вт — анти-дребезг)
  • разрыв данных > 2 с (пропуски не интерполируем — нагрев не «дорисовать»)
  • достигнут максимум 600 с (длинные сессии режем для гранулярных точек)
```

Соответствие требованиям ТЗ и анти-шумовые правила:

| Шум | Механизм подавления |
|---|---|
| Импульс нагрузки < 3 с | не наберёт 15 с в CANDIDATE → окна нет |
| Провал нагрузки < 3 с внутри окна | грейс-счётчик 3 с не размыкает STABLE |
| «Пила» 35↔80 Вт (нестабильная нагрузка) | гейт `CV(P) = σ/μ < 0.20` по окну |
| Не вышли на тепловое плато | гейт `|dT/dt| хвоста < 0.15 °C/с` |
| Одиночные выбросы сенсора температуры | медианный фильтр 5 с при чтении (raw не трогаем) |
| Пропуски телеметрии > 2 с | разрывают окно |
| Физически невозможные значения | обнулены ещё на ingest (флаг) |

Все пороги — в конфиге `analysis_params` с пресетами по `device_class` и
точечными `devices.analysis_overrides`; в коде констант нет.

Окно, упирающееся в правый край прогона, ещё ОТКРЫТО и не эмитится: CV на
обрезанном куске проходит гейт и рождает фантомную точку, которую полные
данные потом бракуют, но перезаписать не могут (другой `window_start`).
Начало открытой сессии персистится в `analytics_state.open_since` — следующий
прогон перечитывает серию от него и воспроизводит чанки с теми же ключами
(окно может пересекать границу многих запусков джоба); ключ `UNIQUE
(device_id, domain, window_start)` делает повторную обработку идемпотентной.
Молчание агента ≥ gap_split перед границей = обычный гэп, окно закрывается;
опоздавший дренаж спула честно пересчитает reprocess (delete+detect отрезка).

### 5.4 Расчёт Rth по окну

Хвост окна = последние `min(10 с, duration/3)` — самая установившаяся часть:

```
Rth(w) = (mean(T_tail) − T_amb(day)) / mean(P_tail)
```

Композитный `quality ∈ [0..1]` (геометрическое среднее субскоров): длительность
окна, `CV(P)`, `|dT/dt|` хвоста, доля пропусков, и ambient-фактор
`0.5 + 0.5·ambient_confidence` — вклад ambient смягчён намеренно (уточнение по
живым данным i9-13900HX): ошибка ambient — общий сдвиг УРОВНЯ всех окон дня,
дневные медианы и наклон тренда она почти не искажает, а полное перемножение
выбивало хорошие окна из тренда навсегда при confidence коротких эпизодов ~0.1.
Точки с `quality < 0.5` в тренд не идут; остальные входят с весом quality.

Страта окна: power-бин (`35–50`, `50–80`, `80+` Вт) × fan-бин (RPM-терцили
устройства). Доминирующий процесс сохраняется — тип нагрузки влияет на
распределение тепла по кристаллу, и сравнивать Cinebench с игрой честно можно
только внутри страты или с оговоркой.

### 5.5 Модель деградации

**Эпохи.** История устройства режется журналом обслуживания на эпохи:
эпоху начинают `kind ∈ {repaste, cleaning, repad, hw_change}` (Шаг 6;
fan_curve_change/undervolt_change эпоху не режут — это смена режима, её ловит
CUSUM). Базлайн и тренд существуют только внутри эпохи — замена пасты легально
«обнуляет» деградацию; в свежей эпохе без окон снапшот health сбрасывается в
`sparse`, а не показывает цифры прошлой эпохи.

**Базлайн.** `Rth_base(epoch, stratum)` = медиана Rth первых 14 дней эпохи при
≥ 30 окнах (иначе ждём данных, health = 'sparse'). Headline-страта — та, где
больше всего окон за 90 дней.

**Текущий уровень.** `Rth_cur(d)` = медиана за скользящие 7 дней.

```
degradation_pct(d) = 100 · (Rth_cur(d) − Rth_base) / Rth_base
```

**Тренд.** По дневным медианам `R̃th(day, stratum)` внутри эпохи, окно 90 дней —
оценщик **Theil–Sen** (робастный к выбросам, без предположений о распределении;
доверительный интервал по методу Сена). Slope публикуется при ≥ 10 днях данных
и ≥ 100 окнах; если CI накрывает ноль — показываем «деградация не
обнаружена», а не шумовой наклон.

**Чейнджпойнты.** CUSUM (k = 0.5σ, h = 5σ) по первым разностям дневных медиан
(σ через MAD): ступенька — одна большая разность, плавный дрейф деградации —
много маленьких ниже порога. Ступенька — это не деградация (она градуальна),
а смена условий: новая кривая вентиляторов, андервольт, чистка. Обнаруженный
чейнджпойнт ⇒ предложение пользователю подтвердить событие обслуживания
(`maintenance_events`: kind `regime_change`, source `changepoint_suggested`,
дедуп ±3 дня) и авто-граница псевдо-эпохи — тренд считается по сегменту после
последней ступеньки, чтобы не смешивать режимы. Базлайн эпохи предложение НЕ
сбрасывает: это делает только подтверждённое пользователем событие.

**Разделение конфаундеров (эвристика честности).** Rth = … + Rth_hs→air(RPM):
- рост Rth **при максимальных RPM** так же, как при средних → деградация
  TIM/контакта (воздух не помогает — проблема до радиатора);
- рост требуемых RPM при той же мощности / рост Rth только в низких fan-бинах →
  вероятна пыль в рёбрах или износ вентилятора.
В `device_health.data_quality` и алертах формулируем как «вероятно паста» /
«вероятно пыль» с явной неуверенностью — это повышает доверие, а не снижает.

**Приоры по типу TIM** (`maintenance_events.tim_type`): классическая паста —
медленный монотонный рост (pump-out); жидкий металл — долго плоско, потом
резко (миграция/высыхание) → чувствительнее к ступенькам; PTM7950 — почти
плоско → строже порог алерта, чтобы не спамить ложняками.

**Прогноз.** Троттлинг наступает, когда при типичной нагрузке
`T_amb_typ + Rth_proj·P_typ ≥ T_throttle` (T_throttle по `cpu_model`, дефолт
95 °C; P_typ — медиана мощности headline-страты; T_amb_typ — медиана за 30
дней). Решаем `Rth_base + slope·Δd = Rth_критич` относительно Δd → дата.
Slope ≤ 0 → «не ожидается». Горизонт публикации ≤ 180 дней — дальше линейная
экстраполяция нечестна.

**Health score (v1, настраиваемо):**

```
score = 100 − 40·clamp(deg%/15, 0, 1)
            − 40·clamp((180 − days_to_throttle)/180, 0, 1)
            − 20·penalty(data_quality)
Зоны: ≥80 зелёная · 50–79 жёлтая («планируйте обслуживание») · <50 красная
```

**Детекция троттлинга** (независимый сигнал, валидирует модель): минуты, где
`cpu_temp ≥ Tjmax−2` при снижающейся мощности → счётчик throttle-минут в день;
рост этого счётчика — это симптом, который пользователь уже чувствует руками.

### 5.6 Оркестрация джобов

| Джоб | Период | Что делает |
|---|---|---|
| `detect_windows` | 5 мин | state machine по новым данным (watermark −2 мин), эмит `rth_windows` |
| `estimate_ambient` | дневной (локальная ночь устройства) | idle-эпизоды → `ambient_estimates` |
| `update_trends` | дневной | дневные медианы, Theil–Sen, CUSUM, `device_health`, прогноз |
| `evaluate_alerts` | дневной, после trends | правила алертов → `alerts` + email |
| `reprocess` | ночной | пересчёт диапазонов из `reprocess_queue` (поздние данные, смена `model_version`) |

Шедулер — APScheduler в отдельном asyncio-процессе (worker). Celery/брокер не
нужны: джобы идемпотентны, переживают пропуск тика, параллелятся по устройствам.

### 5.7 Алерты

| kind | Условие |
|---|---|
| `degradation_warning` | deg% > 8 семь дней подряд **и** CI наклона не накрывает 0 |
| `degradation_critical` | deg% > 18 **или** прогноз троттлинга < 45 дней |
| `dust_suspected` | паттерн «растут RPM при той же мощности» (5.5) |
| `throttling_detected` | ≥ N throttle-минут/день, рост неделя к неделе |
| `ambient_calibration_lost` | 14 дней без валидных idle-эпизодов |
| `device_offline` | last_seen > 48 ч |
| `sensor_degraded` | доля NULL по заявленной capability > 50% за сутки |

Дедуп алертов: один активный алерт типа на устройство, повторная отправка не
чаще раза в 7 дней до ack.

## 6. Бэкенд API (FastAPI)

```
Аутентификация устройств:  Bearer device_token (lookup по prefix, сравнение хэша)
Аутентификация людей:      JWT HS256 (email + bcrypt; секрет THERMAL_JWT_SECRET)

— агент (реализовано, Шаги 2–5):
POST /v1/telemetry                       ← батчи телеметрии (device token)
POST /v1/telemetry/pair                  ← публичное сопряжение: pairing-код →
                                           создание Device + постоянный token

— пользователи / фронтенд (реализовано, Шаг 5):
POST /api/v1/auth/register | /login      → JWT (bcrypt в thread-pool)
GET  /api/v1/devices                     → устройства пользователя
POST /api/v1/devices/pairing-code        → код XXXX-XXXX, TTL 10 мин, одноразовый
GET  /api/v1/devices/{id}/health         → снапшот device_health по доменам
GET  /api/v1/devices/{id}/trend?domain=&stratum=&days=  → дневные Rth (медиана,
                                           p25/p75) + наклон/CI из device_health
GET  /api/v1/devices/{id}/timeseries?bucket=1m|1h&from=&to=  → из CAggs
Тенантность: каждый device-эндпоинт через проверку владения; чужой id → 404.

— обслуживание (реализовано, Шаг 6):
POST /api/v1/devices/{id}/maintenance    ← paste_replacement|dust_cleaning|repad
                                           (+performed_at, notes, tim_type) —
                                           новая эпоха, мгновенный пересчёт health
GET  /api/v1/devices/{id}/maintenance    → журнал (включая CUSUM-предложения)

— дашборд (реализовано, v3):
GET  /api/v1/devices/{id}/current-health → шапка одним запросом: последняя
                                           ambient-оценка + по доменам последняя
                                           честная Rth-точка, score, days_to_critical
GET  /api/v1/devices/{id}/rth-history?domain=&stratum=&days=  → скаттер окон
                                           ≥ per-device гейта качества (days ≤ 90;
                                           длинные горизонты — /trend)
GET  /api/v1/devices/{id}/maintenance-suggestions  → открытые CUSUM-предложения;
                                           подтверждение = POST /maintenance в
                                           ±3 дня (закрывает предложение)

— запланировано (v1.1+):
история ambient-оценок · GET /api/v1/alerts + ack · ротация токенов
```

Горячий путь ingest: Pydantic → auth (LRU-кэш девайса) → диапазонные проверки →
резолв процессов (LRU + upsert) → один `INSERT … SELECT unnest(...) ON CONFLICT
DO NOTHING` + запись `ingest_batches` в той же транзакции (asyncpg, без ORM на
этом пути; CRUD-слой — SQLAlchemy async). Rate limit: token bucket 6
батчей/мин/устройство (burst 12), in-process — при горизонтальном
масштабировании переедет в Redis.

Backpressure: деградация БД → 503 + `Retry-After`, агент буферизует локально.
Данные не теряются молча никогда.

Observability: метрики ingest (батчи/с, dup rate, reject rate, lag),
длительности джобов, last_seen по устройствам; structlog + Prometheus-эндпоинт.

## 8. Реестр ключевых решений и трейд-офф

| Решение | Альтернатива | Почему так |
|---|---|---|
| Сырые 1 Гц данные шлются на сервер, детекция окон серверная | умный агент, шлёт только агрегаты | алгоритмы будут меняться часто; version skew агентов страшнее трафика (~3 КБ gzip/30с) |
| Raw хранится 30 дней, выводы — вечно | хранить raw вечно | продукт — тренды Rth, а не байты; storage-смета ×20 |
| Прямой insert, без брокера | Kafka/Redis Streams | 1000 строк/с; брокер = операционный налог без выгоды; контракт позволяет добавить позже |
| `compress_after = 7д = горизонт поздних данных` | компрессить раньше | вставка в сжатые чанки дорогая; инвариант связывает две константы |
| Theil–Sen + дневные медианы | OLS по точкам | Rth-точки тяжелохвостые и кластеризованные; OLS уедет за выбросами |
| Стратификация по мощности/RPM | одна общая кривая Rth | Rth зависит от рабочей точки; смешение страт даёт ложные тренды |
| Эпохи по журналу обслуживания + CUSUM | один сплошной тренд | replaste/настройки ломают ряд ступенькой; без эпох модель лжёт |
| App-level tenancy, без RLS | RLS сразу | один потребитель БД; RLS добавим при внешних читателях |
| Retention одинаковый для всех тарифов | пер-тенант retention | drop_chunks глобален; DELETE-джобы дороги; тарифицируем фичи |
| APScheduler-worker | Celery + брокер | джобы идемпотентны и переживают пропуски; меньше инфраструктуры |

## 9. Риски

1. **Качество сенсоров LHM** (локали, залипания, переименования после
   обновлений) → `detect-sensors`, `sensor_map` на сервере, флаги, серверная
   валидация диапазонов.
2. **macOS/Apple Silicon**: температура кристалла доступна ограниченно →
   capabilities, честная деградация функциональности, Windows-first запуск.
3. **SNR деградации**: сигнал 0.05–0.3 K/W на фоне шума рабочих точек → страты,
   медианы, Theil–Sen, минимальные объёмы данных до публикации выводов; никогда
   не показывать наклон с CI через ноль.
4. **Конфаундеры** (пыль, кривые вентиляторов, андервольт) → RPM-эвристика,
   CUSUM, журнал обслуживания; формулировки «вероятно X» вместо ложной точности.
5. **Нет idle-периодов** у части пользователей → confidence-механика + UX-подсказка;
   в v2 — управляемая калибровка.

## 10. Roadmap

- **v1 (MVP):** агент Windows, ingest, raw+CAggs+политики, окна, ambient, Rth,
  тренд per-device, JSON API, docker-compose. Критерий: на одном реальном
  ноутбуке за 2–3 недели виден стабильный Rth-ряд с CI.
- **v1.1:** health score, прогноз, алерты + email, журнал обслуживания в UI.
- **v2:** агент macOS, веб-дашборд, **режим управляемой калибровки**
  (ADR-0002: кнопка «Запустить калибровку» в UI → агент запускает встроенный
  CPU-бёрнер фиксированной мощности на ~30 с → эталонная, идеально сравнимая
  между неделями Rth-точка), RPM-нормализация регрессией вместо стратификации.
- **v2+:** когортная аналитика — сравнение Rth с другими экземплярами той же
  модели ноутбука («ваш Rth на 23% хуже медианы по 47 таким же Legion 5 Pro»).
  Сетевой эффект данных — главный SaaS-ров продукта.
